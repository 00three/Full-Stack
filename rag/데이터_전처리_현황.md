# 데이터 전처리 현황 및 개선 방향

> 작성일: 2026-05-25
> 목적: 현재 전처리 파이프라인 문제점 진단 + 이미 수정한 것 + 남은 과제 정리

---

## 1. 현재 전처리 파이프라인 구조

```
크롤러 JSONL 1건 (raw_documents 1 row)
   │
   ├── content_text  (HTML 본문 파싱 결과)
   │
   └── attachment_text  (PDF/DOCX/HWP 추출 결과) ← 제거됨 ✅
          │
          ↓
[1단계] split_inputs() — data_type별 분리
          │
          ↓
[2단계] clean_text() — HTML 잔여 제거 + 공백 정규화
          │
          ↓
[3단계] chunker.split() — RecursiveCharacterTextSplitter
          │     chunk_size=400, overlap=100
          │     separators: ["\n\n", "\n", ". ", " "]
          │
          ↓
[4단계] contextualizer.make_context_prefix() — LLM이 맥락 요약 생성
          │     현재 비활성화 (INGEST_USE_CONTEXT=0)
          │
          ↓
[5단계] make_full_text() — context_prefix + original_text 결합
          │     (meta_prefix는 임베딩에서 제외)
          │
          ↓
[6단계] BGEEmbedder.encode() — 1024차원 dense vector
          │
          ↓
[7단계] 중복 체크 — cosine ≥ 0.95면 기존 chunk에 병합
          │
          ↓
[8단계] documents 테이블 INSERT
```

---

## 3. 각 단계별 상세 현황

### 1단계: split_inputs — 텍스트 분리

**변경 전:**
```python
body = content_text → "body_text"
pdf = attachment_text → "pdf_text"   ← 삭제됨
```

**변경 후 (현재):**
```python
body = content_text → "body_text"    ← 이것만 사용
```

`attachment_text`는 `raw_documents` 테이블에 여전히 저장되지만 (nullable, 역호환), chunking/임베딩 대상에서 제외.

---

### 2단계: clean_text — 텍스트 정제

**현재 처리하는 것:**
- HTML 태그 제거 (BeautifulSoup)
- 제어문자 제거 (\x00~\x1f, \n·\t 제외)
- 공백 2개+ → 1개
- 개행 3개+ → 2개

**처리 안 되는 것 (P-4):**
- "개발도상국 들은" 같은 조사 앞 비정상 공백 → 이건 PDF에서 온 문제이므로 PDF 제거로 대부분 해결
- 그래도 body_text에도 일부 있을 수 있음 → 추가 정제 규칙 검토 가치

---

### 3단계: Recursive 청킹

**현재 설정:**
```python
chunk_size = 400      # 한글 400자 ≈ 영어 150-200 토큰
chunk_overlap = 100   # 인접 chunk 경계 100자 겹침
separators = ["\n\n", "\n", ". ", " "]
```

**문제 (P-1, P-2, P-3):**

| 이슈 | 예시 | 원인 |
|---|---|---|
| 사실 단위 절단 | "사전동의 예외 허용을 2년 연장한다고 밝혔다. [여기서 잘림] 이번 조치는..." | 400자가 짧아서 |
| 한국어 종결어미 | "했다.이번" → `". "`에 매치 안 됨 (공백 없을 때) | separators 한계 |
| 마침표 잔여 | chunk 시작: `. 소셜미디어 플랫폼에서는...` | `. ` 기준으로 자른 뒤 뒷부분에 `.` 남음 |

**개선 후보:**
```python
chunk_size = 600      # 또는 800
chunk_overlap = 150   # 비례 증가
separators = ["\n\n", "\n", "다. ", "요. ", "죠. ", ". ", " "]
```

---

### 4단계: Contextual Retriever

**현재 상태:** `INGEST_USE_CONTEXT=0` → 전체 비활성화. 모든 chunk의 `context_prefix`가 빈 문자열.

**원래 의도:**
- 각 chunk에 "이 chunk가 문서에서 어느 위치/맥락에 해당하는지" LLM이 1-2문장 요약
- 검색 정확도 ↑ (Anthropic 논문: 실패율 49% 감소)

**비활성화 이유 (추정):**
- OpenAI API 키 필요 (gpt-4o-mini 하드코딩)
- chunk마다 LLM 1콜 → 비용·시간 비쌈
- 효과 미검증 상태

**overlap과의 기능 중복 (P-11):**
- overlap 100자 = 인접 chunk와의 local 문맥 보존
- contextual prefix = 문서 전체에서의 global 문맥 요약
- 둘 다 "chunking으로 잃어버린 문맥 복구" 목적 → 효과 일부 겹침
- chunk_size를 키우면(600-800) 자체로 문맥 충분 → contextual 필요성 감소

**검증 실험 필요:**
```
A: chunk 400 + contextual OFF  (현재)
B: chunk 600 + contextual OFF
C: chunk 600 + contextual ON
D: chunk 800 + contextual OFF
→ B와 C 차이가 작으면 contextual 불필요
```

---

### 5단계: full_text 조립

**현재:**
```python
full_text = context_prefix + "\n" + original_text
# context_prefix가 빈 문자열이면 (현재): full_text = original_text
```

**의도된 설계 결정 — meta_prefix 임베딩 제외:**
```python
# [KCC | 2026-03-18 | 소상공인 점포 정보 전송 서비스]
# ↑ 이건 안 넣음. BGE-M3가 source 토큰을 과학습하는 부작용 때문.
```

**trade-off:**
- ✅ cross-source 검색 가능 (KCC 쿼리 → NSP chunk 매칭)
- ❌ date 정보 임베딩에 없음 → 시간 컨텍스트 손실
- ❌ title 정보도 없음 → 보도자료 제목이 중요한 정보인데 누락

---

### 6단계: BGE-M3 임베딩

**현재:** dense vector만 사용 (1024차원)
```python
return_dense=True,
return_sparse=False,    ← 미사용
return_colbert_vecs=False,  ← 미사용
```

**미사용 자원:**
- BGE-M3는 같은 forward pass로 sparse + colbert도 줌 (추가 비용 0)
- sparse vector → 한국어 키워드 매칭에 강함 (tsvector 대체 후보)
- colbert → token-level 정밀 매칭

---

### 7단계: 중복 병합 (DUP_THRESHOLD = 0.95)

**동작:** cosine ≥ 0.95인 기존 chunk 발견 시 INSERT 안 하고 `source_doc_ids`에 병합

**문제 (P-9):** BGE-M3가 비슷한 표현을 0.95+로 잡을 수 있어 실제로는 다른 사실이 합쳐질 위험

**확인 필요:** dev DB에서 병합 비율 측정
```sql
SELECT count(*) FILTER (WHERE jsonb_array_length(source_doc_ids) > 1) AS merged,
       count(*) AS total,
       round(100.0 * count(*) FILTER (WHERE jsonb_array_length(source_doc_ids) > 1) / count(*), 1) AS pct
FROM documents;
```

---

### 8단계: tsvector 인덱스

**현재:** `to_tsvector('simple', full_text)` + GIN 인덱스

**문제 (P-8):**
- `'simple'` = 공백/구두점 기준 분리만 (형태소 X)
- `plainto_tsquery`가 AND로 묶어서 긴 쿼리 매치 거의 0건
- 한국어 조사 붙은 채로 토큰화 → "방통위" ≠ "방송통신위원회"
- **사실상 keyword_search가 작동 안 하는 것으로 추정**

## 4. 개선 우선순위

### Phase A: 즉시 적용 (코드 변경 최소)

| 작업 | 효과 | 상태 |
|---|---|---|
| pdf_text 제거 | 검색 풀 62% 쓰레기 제거 | ✅ 완료 |
| source 정규화 (P-7) | 중복 적재 방지 | 미적용 |

### Phase B: 청킹 개선 (재색인 필요)

| 작업 | 기대 효과 |
|---|---|
| chunk_size 400 → 600 or 800 | 사실 단위 보존, 검색 정밀도 ↑ |
| separators 한국어 추가 (`"다. "`, `"요. "`) | 자연스러운 분할 경계 |
| overlap 비례 증가 (150-200) | 경계 정보 보존 강화 |

### Phase C: 검색 인덱스 개선

| 작업 | 기대 효과 |
|---|---|
| BGE-M3 sparse 활성화 | 키워드 검색 보강 (tsvector 대체) |
| tsvector 대체 또는 한국어 토크나이저 | 고유명사·숫자 매칭 ↑ |
| meta_prefix에 date만 부분 포함 | 시간 컨텍스트 회복 |

### Phase D: Contextual Retriever 검증

| 작업 | 목적 |
|---|---|
| chunk 크기 별 contextual ON/OFF 비교 | 비용 대비 효과 정량화 |
| LLM 모델 변경 (gpt-4o-mini → Haiku) | 비용↓ + Bedrock 통일 |
| rule-based prefix 대안 테스트 | LLM 없이 [출처/제목/위치] 간단 prefix |
