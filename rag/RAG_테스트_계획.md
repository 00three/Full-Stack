# RAG 테스트 계획

> 작성일: 2026-05-24
> 짝 문서: [RAG_개선_계획.md](RAG_개선_계획.md)
> 목적: **같은 입력에 변수만 바꿔 출력 비교** — 변수 효과를 직접 측정

---

## 핵심 아이디어

```
[고정 테스트셋]               [변수 조정]                 [평가]
보도자료 + 참고기사 페어 →   config A로 기사 생성   →   점수
       │                     config B로 기사 생성   →   점수
       │                     config C로 기사 생성   →   점수
       │                                                ↓
       └────────────  같은 입력이라 결과 차이 = 변수 효과
```

**라벨링 X, recall@k X, retrieval 평가 X.** 테스트셋 하나만 잘 만들면 영구 재사용.

---

## 1. 테스트셋 구성 (한 번만 만들면 끝)

| 항목 | 권장 |
|---|---|
| 페어 개수 | **8개** (최소 5개, 충분히 많으면 10개) |
| 페어 1개 | 보도자료 1개 + 사람이 고른 참고기사 chunk 3-5개 |
| 다양성 | 짧은/긴 보도자료, 정책/발표/노조 등 장르 섞기 |
| 출처 분포 | KCC 2-3, NSP 2, MBC 2, NODONG 1-2 |

### 만드는 절차

1. dev DB에서 보도자료 후보 20-30건 추출 (`source`별 골고루)
2. 그 중 8건 사람이 선택
3. 각 보도자료에 대해 baseline 검색 결과 top 20을 봐서 적절한 chunk 3-5개 선정
4. `eval/testset.json`에 저장

### 저장 형식

```json
[
  {
    "test_id": "T01",
    "press_release_id": "kcc_20260321_a3b7c1d2",
    "press_release_title": "방통위, 소상공인 점포 정보 전송 서비스 2년 연장",
    "related_chunk_ids": [
      "kcc_20240315_xx_002",
      "nsp_20250101_yy_001",
      "mbc_20250820_zz_003"
    ],
    "memo": "정책 연장 발표, 과거 도입 시점 참고기사 포함"
  },
  ...
]
```

**소요시간**: 1시간 이내.

## 2. 평가 기준 — 정량 + 정성

### A. 정량 평가 (자동, 스크립트가 처리)

| 지표 | 정의 | 좋은 값 | 측정 방법 |
|---|---|---|---|
| **본문 길이** | 생성된 body 글자수 | 1400-2200자 | `len(body)` |
| **citation 분포** | 출처별 [N] 마커 사용 비율 | [1] 단독 비중 50% 이하 | regex 카운트 |
| **참고기사 어휘 등장률** | 참고기사 본문의 핵심 명사 중 새 기사에 나타난 비율 | 30% 이상 | 명사 추출 후 교집합 |
| **참고기사 인용문 사용** | 참고기사의 따옴표 인용이 본문에 등장 | 1개 이상 | `"..."` 패턴 매칭 |
| **참고기사 숫자 사용** | 참고기사의 숫자(연도·금액·인원)가 본문 인용 | 2개 이상 | 숫자 정규식 + 교집합 |
| **환각 의심 숫자** | 본문엔 있는데 소스 어디에도 없는 숫자 | 0개 | 숫자 차집합 |
| **응답 시간** | 생성 latency (ms) | 모니터링용 | 측정 |

### B. 정성 평가 (사람이 1-5점)

| 항목 | 1점 | 5점 |
|---|---|---|
| **사실 정확성** | 환각 다수, 소스와 모순 | 모든 사실이 소스와 일치 |
| **참고기사 활용도** | 참고기사 내용이 거의 안 보임 | 참고기사가 자연스럽게 녹아있음 |
| **정보 균형** | 보도자료만 그대로 풀어씀 | 보도자료 + 참고기사 적절 배합 |
| **기사 완성도** | 제목·리드·본문 어색 | 완성된 한 편의 기사 |
| **가독성** | 어색한 번역체, 반복 | 자연스러운 한국어 |

8개 페어 × 5개 항목 = **40점 채점**. **1개 config당 15-20분**.

## 3. 실험 진행 절차

### 셋업 (한 번만, ~1.5시간)
- [ ] dev DB 만들고 운영 데이터 복제 (30분)
- [ ] 테스트셋 8개 페어 선정 + `eval/testset.json` 저장 (1시간)
- [ ] 평가 스크립트 준비 (제공 받음)

### 1회차당 (~20분)
```
1. 환경변수/프롬프트 변경 (1분)
2. 생성 스크립트 실행 (5분, LLM 8번 호출)
3. 정량 측정 자동 (1분)
4. 정성 채점 (15분)
5. eval/results.csv에 기록 (1분)
```

### 결과 기록 양식 — `eval/results.csv`

```csv
exp_id,date,params_changed,body_len_avg,citation_diversity,ref_vocab_pct,ref_quote_count,ref_number_count,suspicious_numbers,quality_avg,verdict,notes
EXP-01,2026-05-25,baseline,1820,0.18,22%,1.2,1.5,1.1,3.1,baseline,
EXP-02,2026-05-25,REF_MAX_CHARS=2500,1856,0.34,38%,2.5,3.0,0.8,3.8,KEEP,참고기사 활용 ↑
EXP-03,2026-05-25,+MAX_RELATED=8,1890,0.41,42%,2.8,3.2,0.9,3.9,KEEP,
EXP-04,2026-05-26,+prompt tone,1875,0.45,48%,3.1,3.5,0.7,4.2,KEEP,프롬프트 효과 큼
```

verdict: `KEEP` / `REVERT` / `NEUTRAL` / `MIXED`.

## 4. 실험 회차 계획

| 회차 | 변수 | 기대 |
|---|---|---|
| EXP-01 | (그대로) | baseline 측정 |
| EXP-02 | `LLM_GENERATE_REF_MAX_CHARS=2500` (현재 1200) | 참고기사 본문이 더 들어감 |
| EXP-03 | EXP-02 + `LLM_MAX_RELATED_CHUNKS=8` | 참고기사 더 많이 사용 |
| EXP-04 | EXP-03 + 시스템 프롬프트 톤 조정 ("actively incorporate quotes, concrete details") | 참고기사 능동 활용 유도 |
| EXP-05 | EXP-04 + `LLM_EXTRACT_REF_MAX_CHARS=1500` (현재 700) | JSON 추출 단계부터 참고기사 보존 |
| EXP-06 | EXP-04 + 생성 모델 변경 (Sonnet → Opus) | 추론 품질 향상 검증 |

**6회차 × 20분 = 2시간**.

### 확장 (시간 여유 시)
- EXP-07: 톤 변경 (mediaus / professional / mz)
- EXP-08: extracted_json 구조를 main_facts + context_facts로 분리
- EXP-09: citation 균형 후처리 추가
- EXP-10: 다른 보도자료 도메인(노조 성명)에서도 같은 결과인지 검증

## 5. 제공 받을 스크립트

| 파일 | 역할 |
|---|---|
| `eval/generate_test_articles.py` | 테스트셋 받아서 N개 기사 생성, JSON으로 저장 |
| `eval/measure_quantitative.py` | 생성된 기사들의 정량 지표 자동 측정, CSV 출력 |

## 7. 의사결정 규칙

각 회차 결과 본 뒤:

| 변화 패턴 | 행동 |
|---|---|
| 정량 + 정성 둘 다 ↑ | **KEEP**, 다음 변수로 |
| 정량 ↑ / 정성 ↓ | **MIXED**, 정성 채점 다시 보고 원인 파악 |
| 정량 무변화 / 정성 ↑ | **KEEP** (정성 우선) |
| 둘 다 무변화 | **NEUTRAL**, 변경 안 적용 (복잡도만 ↑) |
| 둘 다 ↓ | **REVERT** |

**중요**: 한 회차에 변수 1개만 바꿉니다. 두 개 동시에 바꾸면 어느 게 효과인지 모름.

## 8. 첫 단계 — 지금 할 일

EC2에 SSH 접속해서 postgres 컨테이너 안에서 SQL 1줄:

```sql
SELECT document_kind, source, count(*)
FROM raw_documents
GROUP BY document_kind, source
ORDER BY document_kind, source;
```

결과로 다음 확인:
- `press_release` 각 source별 50건 이상 있나
- `reference_article`가 충분히 있나

OK면 → dev DB 셋업으로 바로 진행.
부족하면 → 크롤러 한 사이클 더 돌려서 보강 (1-2일 자연 증가).
